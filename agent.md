# IVIC v2 施工准则

本文档是后续施工的执行基线。所有功能、UI、重构和验收必须优先对照：

- `invoice-incase/docs/ivic-v2-srs.tex`
- `invoice-incase/docs/ivic-v2-srs.pdf`
- `invoice-incase/docs/data-structure-improvement.md`
- `invoice-incase/docs/project-overview.md`
- `invoice-incase/misc/new/*.png`
- `invoice-incase/misc/stylealike/*.png`

如果 `invoice_manager/docs` 中存在同步副本，也可参考，但以 `invoice-incase` 下的源材料为准。

## 0. 工作模式

当前项目已经进入验收和返工阶段。除非用户明确说“开始施工”“开始修”“可以改代码”，否则只允许：

- 分析反馈
- 阅读文档
- 阅读代码
- 给出差异清单
- 编写或更新施工说明文档

不得擅自修改业务代码、数据库脚本、UI 文件或运行会改变项目状态的命令。

## 1. 子智能体使用要求

后续每一轮较大施工必须使用子智能体，不能只在第一轮使用。

推荐分工：

- SRS 审阅智能体：只读提取 `ivic-v2-srs` 中本轮相关的需求条款、验收点和不可遗漏事项。
- 效果图审阅智能体：只读审阅 `misc/new` 和 `misc/stylealike`，输出页面布局、视觉风格、交互状态和截图验收点。
- 代码审计智能体：只读检查当前实现和 SRS/效果图的偏差。
- 实施智能体：只有在用户明确允许施工后使用；必须限定文件所有权，避免多个智能体写同一文件。
- 验证智能体：在主线程施工时并行做只读验证、截图比对、行数检查或风险扫描。

使用规范：

- 每轮开始前先写明本轮目标、参考材料、子智能体分工。
- 子智能体任务必须具体、可交付、边界清晰。
- 不得让多个子智能体同时修改同一文件。
- 子智能体返回后，主线程必须汇总差异和决策，不得盲目采纳。
- UI 施工必须至少包含一个 SRS 对照子任务和一个效果图对照子任务。

## 2. IVIC v2 产品定位

IVIC 是个人离线桌面发票管理应用，不是多人财务 SaaS，也不是营销展示页。

核心体验目标：

- 本地化：数据库、发票 PDF、截图、附件、配置均保存在本地。
- 工作台式：用户打开后直接进入可操作界面，而不是欢迎页或宣传页。
- 可扫读：表格、卡片、状态、金额要高密度但清晰。
- 可修正：OCR 和导入结果允许预览、人工编辑、确认后入库。
- 可追踪：发票、订单、批次、子订单、到账交易和附件之间关系清楚。

## 3. 全局 UI 风格

必须遵循 `ivic-v2-srs` 的工作台式 UI 和 `misc/new` 效果图：

- 固定左侧主导航，右侧为工作区。
- 背景使用低饱和暖色或浅灰白，不使用强烈渐变、营销式大 hero、装饰性大图。
- 文字和边框以深棕、墨色、棕灰为主。
- 当前导航项使用浅蓝底色或棕色侧边高亮。
- 面板、卡片、按钮、输入框圆角不得超过 8px。
- 主要容器使用 1px-2px 清晰边框，保留 InCase 风格的“工具感”。
- 表格使用浅色交替行，金额列和状态列突出但不刺眼。
- 禁用按钮必须有明确禁用态，不得隐藏关键操作入口。
- 空状态、异常状态、保存成功、保存失败、导入结果必须有清楚反馈。
- 底部需要窄状态栏，显示数据库/附件/最近操作状态。

禁止：

- 做成营销落地页。
- 用大面积单色紫蓝渐变、装饰气泡、过度卡片化。
- 把所有功能堆进一个巨大页面或巨大弹窗。
- 只换颜色但不改布局。

## 4. 主导航和页面命名

SRS 要求主导航至少包含：

- 首页/总览
- 表单管理
- 提交批次
- 对账表
- 分组管理
- 设置

后续 UI 施工应逐步把现有页面映射到这些模块，而不是继续沿用老式“项目管理/发票管理/报销单管理/到账对账/统计查询”的割裂结构。

允许保留底层业务模块，但用户可见主导航应向 SRS 靠齐。

## 5. 首页/总览要求

参考 `misc/new/mainpage.png`。

必须实现：

- 左侧固定导航和 LOGO。
- 右侧顶部总览区：
  - 分组名或当前范围。
  - 金额统计。
  - 表单/订单/批次/到账数量统计。
  - “跟随/隐藏金额”按钮，可将敏感金额显示为星号。
  - 全部/特定分组切换。
- 右上搜索框。
- 提醒开票区：
  - 展示条目、日期、金额、逾期天数等。
  - 可滚动。
  - 点击条目跳转到表单管理并高亮对应条目。
- 主内容区：
  - 搜索为空时展示统计图像或概览。
  - 搜索有内容时展示搜索结果。
- 底部窄状态栏：
  - 数据库状态。
  - 附件目录状态。
  - 最近操作提示。

## 6. 表单管理要求

参考 `misc/new/sheetmanage.png` 和 `sheetmanagematching.png`。

表单管理是发票、订单、附件、匹配关系的主入口，不应只是普通发票表格。

功能区必须包含：

- 搜索框
- 导入
- 批量导入
- 多选
- 导出
- 提交
- 分组选择
- 内容类型选择
- 状态选择
- 重置
- 匹配
- 编辑
- 删除

按钮启禁规则：

- 导出默认禁用，选中一行或多行后启用。
- 编辑默认禁用，只在单选时启用。
- 删除默认禁用，选中一行或多行后启用，删除前必须确认。
- 提交默认禁用，选中一行或多行后弹出批次提交窗口。
- 匹配进入匹配模式后，普通编辑和双击详情必须禁用。

表格要求：

- 支持单元格直接编辑：选中条目后再次点击单元格进入编辑，回车或点击外部确认。
- 支持双击详情弹窗。
- 支持表头排序。
- 支持合理列宽：重要信息列宽大，不重要信息列宽小。
- 支持浅色交替行。
- 字段至少包含名称/摘要、金额、日期、分组、内容类型、状态、是否有发票、是否匹配、附件数量、备注、更新时间。

匹配模式：

- 表格区切换为上下两个表格。
- 上表：未有发票的订单项目。
- 下表：已有发票的项目。
- 两表均为单选。
- 中间提供 match/unmatch、取消、完成匹配。
- match 后项目只在上表显示，不再显示于下表。
- 临时匹配状态必须用颜色或标记显示。
- 取消丢弃临时变更。
- 完成匹配后写库。

## 7. 导入流程要求

参考 `misc/new/addui.png`、`addui-batch.png`、`addlogic.png`。

单条导入必须包含：

- 订单文本框。
- 发票拖拽/上传框。
- 附件拖拽/上传框。
- 附件备注输入。
- 发票上传后 OCR 结果替换或填充文本区。
- 右侧 PDF 原件预览。
- 提交按钮。

批量导入必须包含：

- 多发票文件上传。
- 总数、总金额、问题数统计。
- 可编辑预览表格。
- 当前选中行对应 PDF 预览。
- 确认导入按钮。

导入规则：

- 无有效内容时提交禁用。
- 如果表中已有内容，导入按钮必须禁用或要求用户确认覆盖/追加策略。
- OCR 不是强制准确来源，必须允许预览和人工修正。
- 确认前不得写入正式表。
- 批量导入不得一次性加载所有 PDF 预览，只能按当前选中行按需加载。

## 8. 提交批次要求

参考 `misc/new/batchsubmit.png`。

必须实现：

- 功能区类似表单管理，但面向批次。
- 表格区以批次为一级行。
- 每个批次行前有可展开符号。
- 点击展开后展示批次内所有订单/子订单。
- 展开符号应有旋转或明确展开态。
- 双击批次或选中后编辑，打开批次编辑窗口。

批次编辑窗口：

- 左侧为可编辑批次详情文本框。
- 右侧为快速提交文本区。
- 快速提交文本区用于复制到外部学校/财务系统。
- 编辑保存前需要确认。

## 9. 对账表要求

参考 `misc/new/incomesheet.png` 和 `incomesheetmatching.png`。

普通状态：

- 功能区包含搜索、新增、多选、导出、分组/类型/状态筛选、重置、对账、编辑、删除。
- 新增到账交易窗口可填写到账金额、交易时间、备注，并上传或拖拽截图附件。
- 表格中交易可展开，展示交易包含的批次。
- 批次可继续展开，展示批次内订单/子订单。

对账状态：

- 点击对账后，页面切换为对账状态，而不是只弹出普通小窗口。
- 部分非对账按钮禁用。
- 上表显示到账交易。
- 下表显示报销批次/订单。
- 支持多选交易、多选批次或订单。
- 左下角显示实时差额：交易金额 - 订单总金额。
- 金额相等时差额模块显示成功态。
- 金额不等允许保存，但必须给相关交易、批次、订单打异常或差异标记。
- 临时选择和修改过的行必须变色。
- 单次对账可以处理多笔交易。
- 取消丢弃临时变更。
- 完成后写库。

## 10. 分组管理要求

参考 `misc/new/groupmanage.png`。

必须实现左右分栏：

- 左侧为分组卡片列表。
- 左侧顶部有搜索框。
- 分组卡片显示分组名称和简要信息。
- 卡片过多时可滚动。
- 列表末尾有虚线或明显样式的“新增分组卡片”。
- 右侧为分组详情编辑区。

分组详情至少包括：

- 分组名称。
- 简介。
- 发票抬头规则。
- 备注。
- 启用状态。

关键要求：

- 发票抬头规则是分组自动建议的重要来源。
- 导入、匹配或录入时应尽量根据抬头规则建议分组。
- 不得把分组管理做成单纯表格页。

## 11. 设置页要求

参考 `misc/new/settings.png`。

设置页必须让用户感觉是在管理本地文件，而不是配置数据库服务器。

必须实现：

- 本地数据库文件路径选择。
- 附件目录路径选择。
- 备份/恢复入口。
- 常规设置，如日夜模式、检测更新开关、开发者信息。

禁止默认暴露：

- 数据库主机。
- 端口。
- 连接串。
- MySQL 用户名密码。

高级数据库配置如保留，必须折叠到开发者/高级区域，默认不出现。

## 12. 附件和本地文件

必须本地化保存：

- 发票原件。
- 银行到账截图。
- 聊天记录截图。
- 财务系统截图。
- 补充材料。

数据库中保存相对路径或可迁移路径。

删除业务对象或附件时：

- 必须提示用户是否处理本地文件。
- 默认不得静默删除原始用户文件。
- 删除应用复制到上传目录中的文件时，也必须确认。

## 13. 代码施工规范

必须遵守：

- 模块化、低耦合、可复用。
- 单个代码文件超过 400 行必须停下来评估拆分。
- 新增 UI 页面优先拆成：
  - `*_page.py`
  - `*_dialogs.py`
  - `*_columns.py`
  - 必要时 `*_widgets.py`
  - 服务层放在 `services/*_service.py`
  - 纯规则放在 `domain/*`
- 不得把几百上千行 UI、SQL、业务逻辑混在一个文件。
- 页面不直接拼复杂 SQL；复杂查询进入 service。
- service 不依赖 PyQt。
- domain 不依赖数据库和 PyQt。
- 数据库 schema 变更必须同步：
  - MySQL 初始化脚本。
  - SQLite 初始化脚本。
  - 运行时兼容迁移。
  - 演示数据或 demo query。
- 任何写库流程必须有错误提示和可回滚策略。
- 施工前读现有代码风格，优先沿用既有工具函数和样式体系。
- 不得回退用户已有改动。

## 14. UI 验收规范

UI 改动不能只跑编译，必须截图验收。

每轮 UI 施工至少验证：

- 桌面尺寸截图。
- 窄窗口截图。
- 页面是否符合 `misc/new` 对应图的布局。
- 左侧导航选中态是否正确。
- 功能区按钮是否按状态启用/禁用。
- 表格列宽是否能扫读核心信息。
- 临时匹配/临时对账是否有颜色标记。
- 空状态、异常状态、成功/失败提示是否可见。
- 弹窗文本是否不溢出。
- 文件行数是否仍低于 400 行警戒线。

关键页面截图对照：

- `mainpage.png` 对首页/总览。
- `sheetmanage.png` 对表单管理。
- `sheetmanagematching.png` 对匹配模式。
- `addui.png` 对单条导入。
- `addui-batch.png` 对批量导入。
- `batchsubmit.png` 对提交批次。
- `incomesheet.png` 对对账表普通状态。
- `incomesheetmatching.png` 对对账状态。
- `groupmanage.png` 对分组管理。
- `settings.png` 对设置。

## 15. 验收清单

后续任何声称“施工完成”的回答前，必须确认：

- 已重新对照 SRS。
- 已重新对照 `misc/new` 效果图。
- 已使用子智能体做至少一次 SRS/UI 审阅。
- 已列出当前实现与 SRS 的差异。
- 已说明哪些差异已修、哪些仍未修。
- 已跑必要编译或测试。
- UI 施工已截图验证。
- 未超过文件长度规范。
- 没有把老布局简单换皮后声称完成。

如果 UI 风格、布局和 SRS 效果图仍不一致，只能说“业务逻辑完成”或“部分完成”，不得说“整体施工完成”。
